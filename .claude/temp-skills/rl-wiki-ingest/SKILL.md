# RL-Wiki — Karpathy LLM Wiki 投研知识库

基于 Andrej Karpathy 的 **LLM Wiki 模式**：不是每次查询时重新 RAG 检索，而是维护一个持久的、不断积累的维基百科层。你负责来源和判断，LLM 负责所有记账工作（更新、交叉引用、保持一致性）。

## 核心架构

```
原始来源 (raw/)              Wiki 层 (wiki/)                模式 (wiki/CLAUDE.md)
─────────────────            ──────────────────            ──────────────────────
不可变的真理来源               LLM 完全拥有的知识层            维基组织规则
文章、论文、财报、公告           结构化 markdown 文件           页面格式约定
PDF、微信文章、研报             互相链接，交叉引用              写入工作流
                               index.md + log.md 导航
```

## 目录结构

```
Vault_共享知识库/
├── raw/                          # 原始来源（不可变，LLM 只读写入源）
│   ├── articles/                 # 微信文章、新闻剪报
│   ├── reports/                  # 研报 PDF、财报
│   ├── announcements/            # 公告、监管文件
│   └── notes/                    # 人工笔记、会议纪要
├── wiki/                         # LLM 维基层（LLM 可读写）
│   ├── CLAUDE.md                 # Schema — 页面格式约定
│   ├── index.md                  # 内容目录（面向内容）
│   ├── log.md                    # 变更日志（按时间顺序，append-only）
│   ├── companies/                # 公司页面 — 单个公司，一页一个文件
│   ├── industries/               # 行业页面 — 行业全景
│   ├── concepts/                 # 概念页面 — 主题、概念、技术
│   ├── portfolio/                # 组合视图
│   ├── summaries/                # 来源摘要 — 原始来源的摘要页面
│   └── answers/                  # 查询归档 — 有价值的分析回答回写
└── (其他业务文件夹)               # 00_公司基础信息, 01_宏观数据 等
```

## 三大工作流

---

### 1. 摄取 (Ingest) — 新来源 → Wiki 页面

**触发词**：
| 触发词 | 示例 |
|--------|------|
| `编译wiki` | "编译wiki" / "编译宁德时代的wiki" |
| `更新wiki` | "更新wiki" / "更新比亚迪的wiki页面" |
| `wiki ingest` | "wiki ingest" |
| `收录/添加到 wiki` | "收录这篇文章到 wiki" |
| `为{公司}生成wiki页面` | "为宁德时代生成wiki页面" |
| **自动触发** | 投研团队 Step 8.5（报告生成后自动调用） |

**流程**：

```
用户: 把这篇来源加入 wiki [链接/文件]
LLM:
  1. 读取原始来源（文件 / WebFetch / wechat-article-fetch）
  2. 保存到 raw/ 目录（若尚未存在）
  3. 提取关键信息：实体、数据、观点、事件
  4. 与现有 wiki 对比：哪些页面需要更新？哪些需要新建？
  5. 创建/更新 wiki 页面（遵循 CLAUDE.md schema）
  6. 更新 index.md 中相关条目
  7. 在 log.md 追加变更记录
  8. 向用户汇报：更新了哪些页面，发现了什么新信息
```

**摄取规则**：
- 一个来源可能涉及 **10-15 个 wiki 页面** 的更新
- 公司类来源：更新 `companies/{公司}.md`，同时更新相关的 `industries/` 和 `concepts/` 页面
- 行业类来源：更新 `industries/{行业}.md`，更新相关公司的投资逻辑
- 事件类来源：可能更新多个公司的风险因素、估值水平
- **矛盾标注**：新数据与旧断言矛盾时，在目标页面中用 `> ⚠️ 与新来源矛盾：...` 标注，**不要直接覆盖旧数据**
- 所有新增或修改的页面必须在末尾更新 `## 7. 最后更新` section
- 必须在 `log.md` 追加记录

**数据来源优先级**：
1. **ifind MCP** — 中国 A 股/港股的首选数据源
2. **ifind 备用脚本** — MCP 限流/失效时
3. **东方财富妙想** — 第二级备用
4. **wechat-article-fetch** — 微信公众号文章
5. **本地 raw/ 文件** — 已下载的 PDF、研报、笔记
6. **网络搜索** — 以上均不可用时

**投研团队 Step 8.5 集成**：
在 HTML 报告生成后，自动调用本 Skill 为当前公司编译/更新 Wiki 页面。
数据来源：刚生成的投研报告 + 已有 Vault 资料 + 雷达数据。

**rl-company-research-model 集成**：
Excel 模型生成后，提取关键财务数据更新 Wiki 的"财务摘要"和"估值水平" Section。

**rl-radar-agent 集成**：
雷达扫描发现重大变化（财报发布、股价异动）后，增量更新 Wiki 的估值和风险因素。

### 2. 查询 (Query) — 针对 Wiki 提问 → 综合回答 → 可选归档

**触发词**：`wiki {公司/行业}` / `查询wiki` / `wiki 查一下` / `知识库中关于...` / `wiki 里怎么说`

**流程**：
```
用户: [提问]
LLM:
  1. 读取 wiki/index.md 定位相关页面
  2. 读取相关 wiki 页面
  3. 综合答案，引用具体页面
  4. 如果答案产生了新的洞察/比较/联系：
     → 询问用户是否归档为新的 wiki 页面
     → 用户同意后写入 wiki/answers/ 或对应类别
     → 更新 index.md + log.md
```

**查询规则**：
- **优先查 wiki**，不要跳过 wiki 直接去搜网络
- 回答时**引用来源页面**：`根据 [[宁德时代_300750.SZ]] 的投资逻辑 section...`
- 如果 wiki 中信息不足，补充搜索后**同时更新 wiki**
- 有价值的比较分析、交叉洞察应归档回 wiki（`wiki/answers/{标题}.md`）

**归档答案格式**：
```markdown
---
title: "{问题/分析标题}"
query: "原始问题"
created_at: "YYYY-MM-DD"
sources: ["[[相关页面1]]", "[[相关页面2]]"]
tags: [分析, 标签]
---

# 标题

## 分析过程
...

## 结论
...

## 交叉引用
→ [[相关页面]]
→ 行业/{相关行业}

## 最后更新
- 日期：YYYY-MM-DD
```

**快捷命令**：
| 命令 | 功能 |
|------|------|
| `wiki index` | 显示 Wiki 主索引 |
| `wiki stats` | 显示覆盖率统计 |
| `wiki recent` | 显示最近更新的页面 |
| `wiki log` | 显示变更日志 |
| `wiki missing` | 显示尚未建立 Wiki 的公司 |
| `wiki stale` | 显示过期数据（超过90天未更新） |

### 3. 清理 (Lint) — Wiki 健康检查

**触发词**：`wiki lint` / `清理 wiki` / `wiki 健康检查` / `检查知识库状态`

**流程**：
```
用户: 清理 wiki
LLM:
  1. 读取所有 wiki 页面
  2. 执行健康检查（见下方检查清单）
  3. 生成健康报告
  4. 自动修复可修复的问题
  5. 列出需要人工介入的问题
```

**健康检查清单**：

| 检查项 | 检测方法 | 修复方式 |
|--------|---------|---------|
| 断裂引用 | wiki 中的 `[[xxx]]` 对应页面不存在 | 删除断裂链接或创建缺失页面 |
| 孤立页面 | 页面没有任何 `[[xxx]]` 指向它 | 在 index.md 中标记为 ⚠️ 孤立 |
| 过期数据 | 最后更新时间超过阈值（见下方） | 标记需要更新，建议数据源 |
| 矛盾断言 | 同一事实在不同页面说法不一致 | 标注矛盾，建议人工判断 |
| 缺失页面 | 正文中频繁提到但无专属页面的概念/公司 | 建议创建新页面 |
| index.md 不同步 | 页面存在但 index.md 未列出 | 自动补全 index.md |
| log.md 缺失 | 页面有更新但 log.md 无记录 | 补记 log |

**数据过期阈值**：

| 数据类型 | 过期阈值 | 检查方式 |
|---------|---------|---------|
| 财务数据 | 90 天 | 对比 `## 7. 最后更新` 日期 |
| 估值数据 | 14 天 | 对比行情日期 |
| 投资逻辑 | 180 天 | 检查是否有新事件 |
| 风险因素 | 60 天 | 检查是否有新风险 |

**清理报告输出**：
```markdown
# Wiki 健康报告 — YYYY-MM-DD

## 概况
- 公司页面: X
- 行业页面: X
- 概念页面: X
- 总页面: X
- 数据来源总数: X

## ✅ 自动修复
- 修复了 N 个断裂引用
- 同步了 index.md（补充 M 个缺失条目）

## ⚠️ 需要关注
- N 个页面数据过期（财务数据）
- M 个孤立页面
- K 处矛盾断言

## 📋 建议操作
1. 更新 {公司} 的财务数据（上次更新 95 天前）
2. 为 {概念} 创建专属页面（被引用 5 次但无页面）
3. 解决 {公司A} 和 {公司B} 关于 {事实} 的矛盾表述
```

## 页面格式约定（Schema）

所有 wiki 页面必须遵循 `wiki/CLAUDE.md` 中定义的 schema。

### 公司页面 — 7 个必填 Section

1. **公司概况** — 一句话定位 + 核心业务 + 代码
2. **投资逻辑** — 3-5 条可验证论点
3. **财务摘要** — 表格：收入、利润、ROE、现金流、毛利率
4. **估值水平** — PE、PB、总市值 + 历史分位
5. **风险因素** — 3-5 条，按严重性排序，标注类型
6. **交叉引用** — 同行、行业、概念
7. **最后更新** — 日期、触发来源、数据新鲜度

### 交叉引用格式

| 类型 | 写法 |
|------|------|
| 同行公司 | `→ [[公司名_代码]]` |
| 行业 | `→ 行业/[[行业名]]` |
| 概念 | `→ 概念/[[概念名]]` |
| 报告 | `→ {日期} {报告名}` |
| 答案归档 | `→ 答案/{标题}` |

### Raw 仓库路径
- **收件箱**：`/Users/zhuang225/Raw 仓库/`

### 支持文件格式
`.md`, `.txt`, `.pdf`, `.doc`, `.docx`, `.url`, `.png`, `.jpg`, `.jpeg`

### 分拣流程

```
Step 1: 扫描 Raw 仓库 所有文件
Step 2: 逐个读取文件内容（PDF/Word 提取文本，图片 OCR 或引用）
Step 3: 识别文件类型 + 识别所属公司/行业
Step 4: 分类移动到 Vault_共享知识库/raw/{type}/
Step 5: 对公司类文件：额外复制到对应公司文件夹
Step 6: 提取关键信息 → 增量更新 Wiki 页面
Step 7: 将已处理的原始文件移到备份文件夹 ~/Documents/Raw-processed/{YYYY-MM-DD}/
Step 8: 生成分拣报告
```

### 分类逻辑

**1. 公司匹配（最高优先级）**
- 使用 48 家持仓公司列表进行匹配（公司名 + 股票代码）
- 如果文件中明确提到某公司 → 归类为该公司相关资料
- 归档路径：`raw/{type}/YYYY-MM-DD_{公司}_{文件名}`
- 额外复制：`Vault_公司基本面Agent/11_公司列表/{拼音首字母}/{公司}_{代码}/研报/`

**2. 行业匹配**
- 如果未匹配到具体公司，但匹配到行业关键词 → 归类为行业资料
- 归档路径：`raw/{type}/YYYY-MM-DD_{行业}_{文件名}`

**3. 无法分类**
- 如果既无公司也无行业关键词 → 归入 `raw/unclassified/`
- **必须提醒用户**：在分拣报告中标注 `⚠️ 未分类文件：{文件名}`，并建议用户手动分类

## 技巧和最佳实践

- **一次摄取一个来源** — 保持参与，检查更新，引导 LLM 强调什么。批量摄取可在低监督下进行。
- **答案回写** — 好的查询答案不应消失在聊天历史中，应归档回 wiki。
- **矛盾标注** — 新旧数据矛盾时不要覆盖旧数据，用 `> ⚠️` 标注矛盾，留待人工判断。
- **Obsidian 图形视图** — 查看 wiki 形状的最佳方式，哪些页面是中心、哪些是孤立。
- **Git 版本控制** — wiki 是 git 仓库，免费获得版本历史和分支。
- **增量更新优先** — 页面已存在时，只更新变化的 Section，保留未变化部分。
